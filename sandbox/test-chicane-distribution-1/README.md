# test-chicane-distribution-1

This sandbox contains the distribution-level regression test for the analytic
`SBEND` C-type chicane placement work.  It is used in two complementary ways:

- as a manufactured OPALX regression with a known first-order longitudinal
  response;
- as an OPALX-vs-OPAL 2022.1 code-to-code comparison for the same hard-edge
  chicane.

The test is intentionally analytic.  It does not use field maps, fringe fields,
or pole-face focusing.  Those effects should be added only after the hard-edge
body-pose and compatibility placement behavior is understood.

## Chicane Model

The chicane is a four-`SBEND` C-type layout with equal bends and two outer
drifts plus one central drift.  The constants are defined in
`manufactured_chicane.py` and mirrored in the input decks.

| quantity | value | meaning |
| --- | ---: | --- |
| `LB` | `1.0 m` | straight bend chord length |
| `TH` | `0.1 rad` | bend angle magnitude |
| `D_CH` | `1.5 m` | outer drift length between bends 1-2 and 3-4 |
| `C_CH` | `2.0 m` | central drift length between bends 2-3 |
| `rho` | `LB / (2 sin(TH/2))` | bend radius from chord and angle |
| `arc_length` | `rho * TH` | hard-edge arc length |
| `R56_ref` | `-TH^2 * (4 LB / 3 + 2 D_CH)` | small-angle target |

For the current constants,

```text
R56_ref = -4.333333333333334e-2 m
```

The manufactured longitudinal target is

```text
z_exit - z_initial = c - R56_ref * delta
```

where `c` is a fitted monitor-frame offset.  The regression checks the fitted
slope and the residual after subtracting the fitted intercept.

## Distribution

The default manufactured distribution has 125 particles:

- five momentum offsets, `delta = {-2e-3, -1e-3, 0, +1e-3, +2e-3}`;
- five initial `z` values from `-1 mm` to `+1 mm`;
- five finite transverse `(x, y)` points;
- zero initial transverse momentum;
- `BCHARGE = 1e-9 C` with `FIELDSOLVER = NONE`;
- zero edge/fringe parameters, `E1 = E2 = HGAP = FINT = 0`.

The finite transverse extent is deliberate.  It verifies that OPALX reads a
non-degenerate bunch from file while keeping the manufactured longitudinal model
dominant.

## Input Decks

| file | role |
| --- | --- |
| `test-chicane-distribution-1.in` | OPALX body-pose chicane using `X/Y/Z/THETA`; this is the primary placement test. |
| `test-chicane-distribution-2.in` | OPALX compatibility deck using old OPAL-style `ELEMEDGE`; this isolates placement from other OPAL-vs-OPALX model differences. |
| `test-chicane-distribution-1_opal.in` | OPAL 2022.1 comparison deck using reference-line `ELEMEDGE` placement. |

`test-chicane-distribution-2.in` is not the target implementation style.  It is
a control case for judging whether observed OPALX-vs-OPAL differences come from
body-pose placement or from other modeling differences.

Run the Python tools with the project virtual environment unless another Python
environment already has `numpy`, `h5py`, `Pillow`, and `matplotlib` installed:


| tool | purpose | main outputs |


The `--info 5` run is useful for checking the loaded distribution bounds before
tracking.  For the default 125-particle distribution it should show finite
initial bounds close to:

```text
MIN R = (-0.10000, -0.08000, -1.00000) mm
MAX R = ( 0.10000,  0.08000,  1.00000) mm
```

```sh

```

## OPALX-vs-OPAL Comparison

Run OPAL 2022.1 and compare the `.stat` files:

```sh
OPAL_PREFIX=/Users/adelmann/OPAL-2022.1 \
  /Users/adelmann/OPAL-2022.1/bin/opal test-chicane-distribution-1_opal.in

/Users/adelmann/.venv-h6/bin/python compare_opal_opalx.py \
  --opalx-stat test-chicane-distribution-1.stat \
  --opal-stat test-chicane-distribution-1_opal.stat \
  --out-dir comparison/opal_code_compare \
  --plots
```

For the OPALX `ELEMEDGE` compatibility control:

```sh
/Users/adelmann/git/opalx/build/src/opalx test-chicane-distribution-2.in

/Users/adelmann/.venv-h6/bin/python compare_opal_opalx.py \
  --opalx-stat test-chicane-distribution-2.stat \
  --opal-stat test-chicane-distribution-1_opal.stat \
  --out-dir comparison/opal_code_compare_elemedge \
  --plots
```

The code-to-code comparison is not expected to be bitwise identical.  The
important interpretation is:

- energy should agree at roundoff level;
- `By_ref` interval shifts expose hard-edge placement/timing differences;
- body-pose vs `ELEMEDGE` differences separate new OPALX placement behavior from
  the old OPAL reference-line model;
- horizontal and longitudinal RMS differences are expected to remain because the
  old OPAL model and OPALX analytic body-pose model are not identical.

## DT Scan

The scan script creates self-contained run directories under
`comparison/dt_scan`.

```sh
/Users/adelmann/.venv-h6/bin/python run_dt_scan.py \
  --case finite zero \
  --dt 1.0e-11 5.0e-12 2.0e-12 \
  --force
```

The relevant scan outputs are:

| directory | content |
| --- | --- |
| `comparison/dt_scan/finite/` | 125-particle finite-transverse-size runs. |
| `comparison/dt_scan/zero/` | 25-particle zero-transverse-size diagnostic runs. |
| `comparison/dt_scan/finite_10000/` | 10000-particle finite distribution run used for the final visual comparison. |

For each time step, `opalx_body`, `opalx_elemedge`, and `opal_2022` contain the
raw runs.  `compare_body` and `compare_elemedge` contain plots and summaries.

zero crossings; edge timing shifts are documented separately in
`by_field_intervals.csv`.

## Current Status

The manufactured OPALX comparison passes the fitted `R56` check for the current
build.  The zero-length temporal monitor can write more than one full-particle
crossing; `compare_results.py` chooses the full-particle monitor step and uses
the intercept-corrected residual for the longitudinal manufactured-solution
check.

For the `dt_5p0em12` OPALX `ELEMEDGE` control, the current fitted values are:

| case | particles | OPALX fitted `R56` [m] | OPAL fitted `R56` [m] |
| --- | ---: | ---: | ---: |
| finite small | 125 | `-4.44581726e-02` | `-4.44415648e-02` |
| finite large | 10000 | `-4.45195920e-02` | `-4.44786273e-02` |

The reference manufactured value is `-4.333333333333334e-2 m`.  The remaining
offset from that value is currently treated as part of the model/measurement
gap to be understood, not as a failure of the plotting or comparison tools.
